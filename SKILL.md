## 模式 P:前置检查(每次 Skill 激活先跑)

**触发**:本 Skill 任何模式被激活时,**第一件事**就是跑这个流程,**不论用户问什么**。

> 目的:用户已选方案 A(对话锚定提醒,2026-04-27 确认),所以每次见面时主动 nag 一次,提醒该看数据了。**不要烦人** — 提醒后立即继续用户原本的请求。

### 步骤
1. **读** `references/state.json`
2. **算** 距 `last_daily_check` 的小时数 H_d、距 `last_weekly_review` 的天数 D_w、当天是否周日(JST)
3. **判定** 是否需要主动提醒:
   - 若 H_d ≥ 24(或 `last_daily_check == null`)→ 标记 `daily_due = true`
   - 若 D_w ≥ 7(或 `last_weekly_review == null`)或当天是周日 → 标记 `weekly_due = true`
4. **输出**(在响应用户原始请求**之前**插入一个简短的提醒块):
   ```
   📌 [对话锚定提醒]
   - 今日数据已 X 小时未查 — 输入「tts今天数据」立即扫描
   - 距上次周度复盘已 Y 天 — 输入「tts本周复盘」立即复盘
   ```
   - 两个都不 due → **完全跳过提醒**,直接处理用户请求
   - 提醒措辞要轻,不要打断用户思路
5. **接着**处理用户原本的请求(写脚本 / 看数据 / 喂爆款脚本)

### 状态更新
- 每次成功跑完模式 B → 写 `last_daily_check = ISO今天`
- 每次成功跑完模式 C → 写 `last_weekly_review = ISO今天`
- 每次成功跑完模式 D → 写 `last_winning_ingest = ISO今天` 并 `ingested_winners_count += N`
- 每次成功跑完模式 A → **追加** 6 维度元数据到 `weekly_output_log`(R5,2026-04-30)
- 模式 P 检测到 JST 周一 → **清空** `weekly_output_log`,归档到 `weekly_output_history`(R5,2026-04-30)

## 模式 A:写脚本(三阶段交互)

> **🎯 核心理念(凌驾于本模式的所有具体规则)**:本 skill 存在的目的 = **产出更多"卖得出货"的脚本(扩充弹药库 / 探索新角度),不是反复覆写、收敛到上周表现最好的那一条**。探索 > 收敛。读 winning-patterns 是为了**把它当弹药箱、主动汲取多样的角度来扩充选择**,而不是优先抄排行榜第一名。每多一条爆款 = 多一个可选项,不是让所有脚本向它靠拢。

**进入前必须先读 3 个弹药池**(2026-06-02 v1.7.0 起):
1. `references/hooks-library.md` — 0-3s 钩子专属库(强制从此调取)
2. `references/cta-library.md` — 13-20s CTA 专属库(强制从此调取)
3. `references/winning-patterns.md` — 爆款档案(理解各种已验证的打法 / 切入点 / 横断メタパターン,用来扩展本次脚本的角度多样性 — 不要默认收敛到 top 模式)

### Step 1:确认核心诉求
问用户:"今天想生成哪种【核心诉求】的脚本?"

1. 效果诉求型 (Effect-Oriented) — 种草 / 唤起需求
2. 价格诉求型 (Price-Oriented) — 击碎决策防御
3. 在库告急型 (Urgency-Oriented) — 临门一脚收割

### Step 2:确认切入点(Angle)
按用户上一步选择,从 `references/formulas.md` 中列出对应的 2~4 个子切入点供选择。

### Step 3:收集 5 个产品变量
1. 品名 (Product)
2. 核心成分 / 物理卖点 (Key Ingredient)
3. 锚点价格 (Anchor Price,如 Amazon 3,500日元)
4. 目标活动价 (Target Price,如 TikTok 2,000日元)
5. 目标受众 (Target,如 30 代颈纹女性)

### Step 4:生成脚本(严格输出格式 — R12 偏好,2026-05-06 / v1.3.1 修订)
**按以下三块顺序输出,第一块独立成块,前后留空行**:

1. **[纯文本脚本 — TTS 适配格式]** — **无标点 + 短行断句(6–14 字 / 行)**;原句号 / 顿号 / 连接处 → 换行;数字 / 字母保留
2. **[封面大字报 + 视觉指导]** — 主标 / 副标 / 贴纸 / 镜头动作 / 字幕动效(此处可用标点)
3. **[R5/R8/Quality Gate 自检 + 转化点解析]** — 引用了 winning-patterns.md / formulas.md 哪条最新模式 + 5 维度差分

**强制输出规则**(违反任何一条都重写):

**A. 基础规则(85 分对标线)**
1. 100% 纯 Tame-guchi(闺蜜语)。**严禁 です/ます**
2. 句子 8–12 字。多用「。」「!」和换行
3. 情绪词(マジで / ガチで / えぐい / ヤバい)按需使用,**不强制**(参照 #1 #3 都没用)
4. 拟声词(パンッ / うるうる / ツヤツヤ / ヒリヒリ / すーっと / もっちり)按肤感选,**至少 1 个**
5. 结构:`[视觉阻断钩子] → [核心诉求展开] → [物理/背书说服] → [偷感 CTA]`
6. **【钩子:强制从 hooks-library.md 调取】(2026-06-02 v1.7.0 重构)** 候选脚本的钩子**必须有匹配的 HOOK-ID**。不在库的钩子需先用模式 F 入库。
   - **生成前**:读 [`references/hooks-library.md`](references/hooks-library.md),按"适用诉求"过滤候选钩子
   - **多候选要素多样化**(R9 / [[feedback_corpus_variety]]):N 条候选脚本的钩子**必须来自不同 HOOK-ID**,严禁所有候选用同一 HOOK
   - **验证状态优先级**:✅✅✅ > ✅✅ > ✅ > ⭐ > 🔍 候補 > 👀 観察期。低数据条目仅在用户明确探索时使用

    - ❌ 不卖惨 / 不强制情感反応(显得操控)

36. **【framing 互斥规则】(R29,2026-05-07 落盘 — CRITICAL)** 不同情感钩之间存在心理脚本互斥:
    | Framing | 主导脚本 | 互斥的元素 |
    |---|---|---|
    | 自分へのご褒美 | 「私は値する」 | 价格 / 死线 / お得感 |
    | 節約 / 賢い買い物 | 「损しない」 | ご褒美 / 自分を大切に |
    | 緊急 / 死线 | FOMO | ゆとり / 余裕 |
    | 教育 / 信任建立 | 「学ぶ」 | 急かし / 強引 |
    | 感官 ASMR | 共鳴 / 享受 | 効能列挙 / 数字証明 |
    **判定原则**:一条脚本**只走一个 framing**。两个叠加 → 心理脚本互相杀伤。
    **节点重叠例外**:节点死线钩 + 节点情感钩(母の日 + セール)看似可叠加,但**必须以一个为主**。主走情感 → 价格 / 死线全部退出语料;主走死线 → 情感元素只做点缀。
    **自检触发**:写完后问 — 这条脚本的"主情感"是什么?除主情感之外有没有其他元素在偷偷消解它?
    **违反时处置**:用户指出 framing 矛盾 → 立即承认,**不为"信息全面"辩护**。

    **R29 補足(2026-05-18 #8 由来):漏斗递进は並列叠加と区別、例外として許可**

    R29 並列叠加禁止は依然有効。ただし **漏斗递进 framing は例外として許可**:

    ```
    教育(あなたは損してる)
    ↓
    解答(正解はこう)
    ↓
    価格(解薬としての値下げ)
    ↓
    死線(限定期間)
    ```

    **判定基準**:
    - ✅ 漏斗:各 framing が**因果鎖**で繋がる(AだからB、BだからC)
    - ❌ 並列:各 framing が**独立に主張**してる(AもBもCも)

    反例:`自分へのご褒美 + 半額以下』(並列、互斥)
    正例:`夜だけは損 → 朝も夜も正解 → 半額近く → 10日まで』(漏斗、因果)

    **実証**:#8 梨花 ビタテクト 5/11 5 単爆款で漏斗式 4 段 framing が爆款を出す。教育型(C 反直覚損失)で特に強力。

    - ❌ 「SNS 热词 ≥1」作为强制项
    - ❌ 「拟声词集中爆发」作为加分项

    **自检触发**(每条脚本写完必跑):
    1. 这句日语,日本人会真的这样说吗?
    2. 这条脚本读起来像「日本人写的」还是「翻译来的」?

38. **【時間性副詞「最近」で鮮度シグナル】(R34,2026-05-15 #7 落盘)** **20-30代 IP / トレンド商品**で、「最近」+ 動詞テンプレを使い鮮度感を投入。

    **理由**:「最近」= 鮮度シグナル = 「今旬」「ホット」感を 1 語で投入。視聴者に「乗り遅れる前にチェックすべき」と思わせる。

39. **【強調語去重 — 同条内 ≤ 2 回】(R36,2026-05-15 #7 落盘)** SNS 強調語(`マジで』 / `ガチで』 / `本当に』 / `えぐい』 / `ヤバい』)は**同一脚本内 2 回以下**。

40. **【真出单 vs 自評区別】(R38,2026-05-18 #8 由来 — CRITICAL)** winning-patterns.md 排行榜・単体档案・state.json で **「真出单(実投放 >3 単)」と「用户喂入評価 / Skill 生成 / 修正版」を必ず区別**する。

    - 排行榜列に「**真出单**」列を追加(本 PATCH で済、#8 後の v1.7.0 で実装)
    - **規則の信頼度判定**:真出单データに最大重み、Skill 自評・用户喂入評価は補助

    **理由**:Skill 内自評循環(Skill が書く → Skill が高評 → Skill が真理化)防止。R30 本地化優先と並ぶ"認知バイアス対策"規則。

    **R38 補足(2026-05-23 #11 由来):真戦由来データの三カテゴリ分類**

    従来の「真出单 vs 自評」二分法では捉えきれない中間データ(投放したが <3 単、ただし CTOR / 完播は強)を「真戦良作」として別カテゴリで保護する:

    | カテゴリ | 判定基準 | 排行榜列 | 規則昇格重み |
    |---|---|---|---|
    | **真出单爆款** | 出单 >3 単 + 投放 ≥ 3 日 | 「真出单」 | 強(最大重み) |

    - **絶対出单数だけで「成功 vs 失敗」に二分するな**。同一商品 / 同一週内でも投放日数差 24h × 流量差 × 算法分発で出单数は揺れる

    - **規則昇格基準の更新**:真出单 ≥2 件 OR **真出单 1 + 真戦良作 ≥1 件**で「強規則」昇格可

    **反例案例(2026-05-23 自己訂正)**:

### 注意事项

- **不重复**:追加前先扫描库里是否有完全相同或近似的条目。有则提示用户,询问是更新现有条目还是仍要新增
- **元数据完整性**:若关键字段(类型 / 适用诉求)无法推断,**主动询问** — 这是新弹药的入档环节,不能糊
- **跨库识别**:用户说「钩子」但提供的是 CTA 风格的句子(或反之),提醒用户确认类别
- **版本注释**:在追加的条目中标 `**添加日期**: YYYY-MM-DD(用户口头追加)』 便于后续追溯

## 模式 E:用户给单条脚本评分(轻量训练信号)

**触发**:用户对刚生成的脚本说"这条 75 分" / "70 分,钩子弱了"等。

### 步骤
1. 在 `winning-patterns.md` 末尾的「评分日志」section 追加一行:
   ```
   - {YYYY-MM-DD} | 商品 | 诉求.子类 | 评分 X | 用户备注
   ```
2. 若评分 ≥ 85 → 视为"准爆款",触发 L1 沉淀(但标 `[未上线验证]`)
3. 若评分 ≤ 60 → 触发反思:"哪一条规则导致低分?要不要降权?"

> 评分体系是补充信号 — 真实爆款数据(出单>3)始终是最强权重,评分次之。

## 安全规则

- **L1 进化层**(`winning-patterns.md`):**自动累积追加**已永久授权(2026-04-27)。无需确认。永不删除旧条目 — 爆款越积越多,横断メタパターンが浮かび上がる设计。
- **L2 公式层**(`formulas.md`):**生成 patch 草案 → 用户 ✅/❌ → 才执行**。新角度需累计 3 次以上爆款验证才提议升级。
- **L3 规则层**(`SKILL.md` 自身):**生成 patch 草案 → 用户 ✅/❌ → 才执行**。仅在 Quality Gate 漏判 / 输出格式失误时提议。
- 若沉淀逻辑判断有歧义,在覆写前可输出"待沉淀草稿"让用户瞄一眼,但不阻塞。
- L2/L3 patch 必须以 diff 形式呈现(改前 vs 改后),便于审核。
